  • HTTP is the application layer protocol that allows web browsers to make requests of web servers and web servers to respond to those requests with web pages and media like video, audio, images, etc.
  • HTTP requires that it is always the 'client' that makes a request of the 'server', never the other way around.
  • When a request is made it travels from the client to the sever, passing through many intermediary machines. Intermediaries at the application layer are collectively called proxies.
  • An HTTP request contains 4 parts:
  • 1. The start line, which in a request includes the verb or noun, GET, POST, PUT, DELETE, HEAD, OPTION, CONNECT, the URI, and the protocol and version being used, e.g. HTTP/1.1
  • For example: GET /xml HTTP/1.1
  • 2. The headers, each of which is on its own line. The are many optional headers. Types of headers include General Headers, Request Headers, and Entity Headers.
  • 3. A blank line that separates the headers from the payload
  • 4. An optional body. This is only present in certain types of requests, e.g. a form.
  • An HTTP response also has 4 parts:
  • 1. A status line with a protocol and version, a status code like 200, 404, 302, etc., and status text.
  • For example: HTTP/1.1 404 Not Found
  • 2. Headers. Response headers can be grouped into General Headers, Response Headers, and Entity Headers. These headers don't have to appear in that order. You might see
  • 3. A blank link
  • 4. An optional body.
  • Problems with HTTP/1.1.
  • The Headers are not compressed, although the body is.
  • The Headers are repeated from message to message, even when they are identical or very similar
  • Multiplexing is not possible. That is, steams cannot be combined.
  • HTTP2 solves these problems by adding an extra step in which messages are turned into frames which are embedded in a stream. Data and header frames are separated. Streams can be combined
